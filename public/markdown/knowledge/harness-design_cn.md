# Harness 设计

Harness 可以理解为模型外部的“工作装置”。模型提供推理、语言和代码能力，但这些能力本身还不能稳定完成复杂任务。Harness 负责把模型放进一个可操作、可观察、可验证的运行环境里，让它能够读写文件、调用工具、执行命令、管理上下文、接受反馈，并在失败后继续修正。

一句话概括：**Agent = Model + Harness。模型包含智能，Harness 让智能变得可用。**

## 为什么需要 Harness

通用模型有几个天然限制：

- 没有持久状态，无法天然记住项目历史。
- 无法直接操作真实环境，需要工具接口。
- 输出是近似的，需要测试、预览和验证。
- 长任务会出现上下文衰减、提前收尾或重复循环。
- 对安全边界、权限和副作用缺乏稳定感知。

Harness 的设计就是从这些限制反推组件。不是先问“我要加什么功能”，而是问“模型为什么会失败，以及什么外部结构能减少这种失败”。

## 常见组件

| 组件 | 作用 | 解决的问题 |
| --- | --- | --- |
| 文件系统 | 保存状态、计划、结果和中间产物 | 无持久记忆 |
| Git / worktree | 记录变更、隔离任务、支持回滚 | 修改不可追踪 |
| Shell / 代码执行 | 运行测试、脚本、检查工具 | 只能说不能做 |
| 沙箱 | 限制权限、隔离风险、提供默认环境 | 工具副作用不可控 |
| Planner | 把模糊需求拆成规格和步骤 | 任务范围低估 |
| Evaluator | 独立评估结果质量 | 自我评估偏差 |
| Middleware | 在模型调用前后注入规则或检查 | 行为缺少运行时约束 |
| Trace / 日志 | 记录决策、工具调用和失败模式 | 无法定位问题 |

这些组件不必一次性全上。好的 Harness 不是越复杂越好，而是在当前模型能力边界外，补上最少但关键的结构。

## Scaffolding 与 Harness

可以把系统拆成两个阶段：

- **Scaffolding**：第一次调用模型前的组装。比如系统提示词、工具 schema、可用技能、角色说明和环境信息。
- **Harness**：第一次调用模型后的运行时编排。比如工具调用、上下文压缩、安全检查、会话恢复、验证和多智能体协调。

这个区分很重要。Scaffolding 决定智能体“带着什么出发”，Harness 决定它“如何持续工作”。前者更像初始化，后者更像运行系统。

## 反馈回路是核心

Harness 设计的关键不是“让模型多做一步”，而是让模型进入可纠错循环。典型回路包括：

1. 生成计划。
2. 执行实现。
3. 运行测试或预览。
4. 根据失败信息修正。
5. 记录结论和遗留问题。

如果没有这个回路，模型输出很容易停在“看起来合理”。一旦加入测试、截图、类型检查、用户验收和日志分析，输出才会逐渐接近“真的可用”。

## Evaluator 的价值边界

Evaluator 不是固定必需品。它的价值取决于任务难度和模型能力的相对位置：

- 任务在模型能力范围内时，Evaluator 可能只是额外开销。
- 任务略超出模型能力时，Evaluator 能显著提升质量。
- 任务远超能力边界时，Evaluator 也可能只能指出问题，无法推动收敛。

因此 Evaluator 应该被当作一种可调节组件，而不是信仰。模型升级后，原本复杂的 Planner、Evaluator 或上下文重置可能需要被删掉，否则 Harness 自身会成为负担。

## Middleware 的价值

Middleware 是 Harness 中非常实用的一层，因为它可以在模型行为周围加运行时约束：

- 任务结束前强制检查是否运行测试。
- 启动时注入目录结构、项目约定和可用工具。
- 检测重复编辑同一文件的循环，及时打断。
- 在上下文接近上限时触发压缩或工具输出卸载。

这类机制的好处是稳定、低成本、可复用。它把“希望模型记住”变成“系统一定会执行”。

## 好 Harness 的判断标准

一个好 Harness 至少满足四点：

1. **可观察**：人类能看到智能体做了什么、为什么做、卡在哪里。
2. **可恢复**：任务中断后可以靠计划、文件和日志继续。
3. **可验证**：输出能通过测试、截图、lint、人工 review 或评分标准检查。
4. **可简化**：当模型能力增强或任务变简单时，可以移除不必要组件。

## 我的理解

Harness 设计不是给模型“加外挂”，而是给模型“搭工作台”。工作台越适合任务，模型越像工程师；工作台越混乱，模型越像聊天机器人。

在个人 AI Building 工作流中，Harness 的意义是把 LLM 从灵感工具变成稳定生产系统：让它能读知识库、加载 Skills、执行 CLI、调用 MCP、写计划、跑验证，并把经验回填到下一次任务里。真正值得积累的不是某一次对话，而是这套能不断改善的工作装置。
