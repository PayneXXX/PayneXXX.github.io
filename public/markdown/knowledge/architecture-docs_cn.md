# 架构文档设计

架构文档不是给项目写一本厚说明书，而是为人类和智能体提供“方向感”。在复杂代码库中，真正耗时的往往不是写补丁，而是弄清楚应该在哪里改、哪些边界不能碰、哪些规则是项目长期成立的基础。

面向智能体的架构文档尤其重要，因为智能体只能使用它能读到的上下文。没有写进仓库的知识，对它来说就等于不存在。

## 三类核心文档

| 文档 | 核心问题 | 作用 |
| --- | --- | --- |
| ARCHITECTURE.md | 做某件事的代码在哪里？ | 建立项目心智地图 |
| AGENTS.md | 智能体该如何在这个仓库工作？ | 给出入口、规则和导航 |
| PLANS.md / ExecPlan | 这次任务要如何推进？ | 记录计划、进度、决策和验收 |

这三类文档形成一个层次：ARCHITECTURE.md 讲系统是什么样，AGENTS.md 讲如何与系统协作，PLANS.md 讲这次要把系统变成什么样。

## ARCHITECTURE.md：心智地图

ARCHITECTURE.md 最重要的任务是回答两个问题：

1. 如果我要做 X，应该去哪里看？
2. 这个系统有哪些长期不变的边界？

它不应该记录每天变化的细节，而应该记录比较稳定的结构，例如模块分层、数据流、关键抽象、外部依赖和架构不变式。

一个好的架构文档通常很短，但能显著减少搜索成本。它应该命名重要实体，说明为什么系统这样组织，并指出什么不应该发生。比如“UI 层不能直接访问数据库”“这个模块不应该依赖运行时配置”“某类状态只能由单一入口更新”。

## AGENTS.md：智能体入口

AGENTS.md 不是把所有规则塞进一个超长文件，而是给智能体一个稳定入口。它应该像目录一样，引导智能体按需阅读更深层文档。

好的 AGENTS.md 具有几个特点：

- 简短，避免一开始占满上下文。
- 明确常用命令、测试方法和项目约定。
- 指向更细的文档，比如 UI 规则、API 规则、数据模型、部署说明。
- 告诉智能体哪些文件谨慎修改，哪些命令必须验证。
- 使用渐进式披露，让信息按需加载。

核心原则是：给地图，不给冗长说明书。

## PLANS.md / ExecPlan：活文档

PLANS.md 或 ExecPlan 是任务级文档。它既是设计规格，也是执行过程记录。对于多小时任务、跨上下文任务或需要恢复的任务，它比聊天记录可靠得多。

一个完整的 ExecPlan 应该包含：

- 目标和用户可见结果。
- 当前进度，最好有时间戳和 checklist。
- 意外发现、约束和决策记录。
- 仓库现状与关键文件路径。
- 后续工作步骤。
- 具体命令、预期输出和验收标准。
- 回滚和恢复策略。

它的价值在于：任何新智能体读完后，都应该能继续执行，不需要依赖原始对话。

## 共同原则

三类文档有共同原则：

1. **稳定优先**：写长期有效的信息，少写容易过期的细节。
2. **导航优先**：文档首先帮助定位，再帮助理解。
3. **渐进式披露**：入口小而清晰，深层信息按需读取。
4. **不变式优先**：说明系统边界和禁止事项，比罗列现状更有价值。
5. **可验证优先**：能被测试、lint 或 review 检查的规则，比口头规则更可靠。

## 与个人网站的关系

这个网站本身也可以按这套方式组织：

- README 负责让后来者理解项目如何运行和部署。
- CONTENT_INTAKE.md 负责收集内容替换信息。
- Knowledge 模块负责展示公开知识体系。
- 项目详情页负责记录每个项目的目标、方法和状态。

当内容越来越多时，网站不只是作品集，也会变成一个个人知识和项目系统的入口。架构文档设计的思想能帮助它保持可维护，而不是变成一堆无法整理的页面。

## 我的理解

架构文档的本质是降低进入成本。对人类是降低理解成本，对智能体是降低上下文成本。写得好的文档会让系统变得更可合作：新人能上手，智能体能导航，未来的自己也能快速恢复记忆。

所以我更愿意把文档看作工程基础设施，而不是附属说明。它不是完成项目后补上的东西，而是让项目能够持续演进的结构。
