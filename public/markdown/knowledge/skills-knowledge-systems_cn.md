# Skills 与知识系统

个人知识库、LLM Wiki 和 Claude Skills 指向同一个方向：把零散经验整理成可复用、可检索、可维护、可调用的能力单元。

传统知识管理解决的是“我如何记住和理解”；面向智能体的知识系统进一步追问：“这些知识如何进入下一次任务执行”。如果知识不能被检索、加载、引用和更新，它就很难真正参与 AI Building 工作流。

## LLM Wiki：知识作为复利工件

LLM Wiki 的关键思想是：Wiki 是持续复利的知识工件。与普通 RAG 不同，RAG 往往在每次查询时重新检索和综合；Wiki 则把知识编译成结构化页面，并在后续不断更新。

可以把系统分成三层：

| 层级 | 内容 | 所有权 |
| --- | --- | --- |
| Raw Sources | 原始文章、论文、图片、数据 | 人类策展 |
| Wiki | 摘要、概念页、实体页、综合分析 | LLM 维护 |
| Schema | 结构、命名约定、工作流规则 | 人类与 LLM 共同演进 |

人类负责选择有价值的资料和提出好问题；LLM 负责摘要、交叉引用、归档、索引和健康检查。它把维护知识库中最繁琐的部分自动化，让知识库有机会长期存活。

## 三种基本操作

### Ingest

摄取新资料时，LLM 阅读来源，写摘要，更新索引，补充相关概念页，并记录变更日志。一次摄取不只是新增一页，而是让新知识进入已有网络。

### Query

查询不是简单搜索，而是读取相关页面后综合回答。更重要的是，好的查询结果可以回填为新的 Wiki 页面，让探索本身也产生复利。

### Lint

健康检查负责发现矛盾、过时信息、孤儿页面、缺失链接和结构混乱。知识库变大后，Lint 是防止它腐烂的关键。

## Index + Log

一个实用知识系统至少需要两个导航文件：

- **Index**：按主题组织所有页面，帮助快速定位。
- **Log**：按时间记录操作，提供知识库演化轨迹。

Index 面向内容，Log 面向时间。前者帮助查询，后者帮助追踪变化。对于中等规模的个人知识库，这种结构往往比复杂向量数据库更透明、更容易维护。

## Claude Skills：知识作为能力单元

Skills 可以理解为轻量级 Harness 或专业知识包。一个 Skill 通常包含：

- `skill.md`：核心流程、触发条件和输出要求。
- 参考文件：示例、风格指南、背景资料、MCP 指令。
- 代码或脚本：自动化处理步骤。
- 非文本资产：图片、模板、演示文件等。

它的关键机制是渐进式披露：记忆层只存技能名称和描述，触发后才加载完整内容。这样单个代理可以管理大量技能，而不会一开始就撑爆上下文。

## 构建 Skill 的方法

一个好 Skill 至少应该回答：

1. 什么时候触发？
2. 输入是什么？
3. 工作流程分几步？
4. 输出格式是什么？
5. 需要哪些参考资料？
6. 哪些地方必须询问人类？
7. 常见错误如何避免？
8. 如何根据使用结果持续更新？

维护时可以按问题类型分流：

| 问题类型 | 处理方式 |
| --- | --- |
| 流程偏差 | 精简核心流程 |
| 背景不足 | 补充参考文件 |
| 根本性错误 | 更新知识库或规则 |
| 工具调用困难 | 增加 MCP / CLI 说明 |
| 好结果出现 | 保存为示例 |

## 从知识库到工作流

知识系统可以分成三个阶段：

1. **可读**：人类能通过 Markdown、图谱、标签理解内容。
2. **可检索**：LLM 能通过索引、搜索、结构化标题找到内容。
3. **可执行**：知识被整理成 Skills、命令、模板和检查清单，能进入任务流程。

很多知识库停在第一阶段。AI Building 的目标是走到第三阶段：让知识不只是“我知道”，而是“系统会用”。

## 我的理解

Skills 与知识系统的核心价值，是把个人经验从一次性消耗变成可复用资产。读一篇文章、做一个项目、踩一个坑，如果最后只存在聊天记录里，很快就会消失；如果被整理进 Wiki、连接到概念、沉淀成 Skill，它就能在下一次任务中再次出现。

这也是我想构建“面向人类”和“面向 Agent”两套知识库的原因：人类需要叙事、理解和图谱；Agent 需要入口、规则、步骤和可调用资产。两者相互连接，才是个人 AI 工作流真正产生复利的地方。
