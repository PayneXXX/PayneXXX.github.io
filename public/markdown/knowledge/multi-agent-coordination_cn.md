# 多智能体协调

多智能体系统的意义不是“把更多模型堆在一起”，而是把复杂任务拆成不同角色、不同上下文边界和不同信息流。单个智能体适合连续执行；多个智能体适合并行探索、独立评估、分工处理和长期协作。

关键问题是：**上下文边界画在哪里，信息如何跨边界流动。**

## 五种基本模式

### Generator-Verifier

一个智能体生成结果，另一个智能体验证质量并反馈修改意见。这种模式适合质量要求高、评估标准可以显式化的任务，比如 UI 设计、代码审查、文案打磨、测试补全。

风险在于 Verifier 如果没有清晰标准，就会变成橡皮图章。因此必须提供评分维度、负面案例、验收标准或可运行检查。

### Orchestrator-Subagent

一个主智能体负责任务拆解、委派和综合，子智能体负责具体子任务。这是最常见也最稳妥的模式，适合代码库调研、模块级实现、批量文档整理等。

风险在于 Orchestrator 会成为信息瓶颈。如果子任务之间依赖很强，主智能体需要不断同步状态，协调成本会迅速上升。

### Agent Teams

多个持久 worker 长时间负责不同领域，保留各自上下文。例如一个负责前端，一个负责后端，一个负责测试，一个负责文档。

这种模式适合持续多步任务，尤其是迁移、重构、研究型工作。风险是共享资源冲突和完成状态难判断，因此需要明确文件所有权、同步规则和合并机制。

### Message Bus

智能体不直接层级委派，而是通过事件总线发布和订阅任务。某个事件触发后，对应 worker 自动处理。

这种模式适合事件涌现型工作流，比如持续监控、自动分类、异步数据处理。风险是系统可观测性变差，消息路由错误可能静默失败。

### Shared State

多个智能体读写同一个共享存储，例如研究数据库、知识图谱、任务看板或笔记库。它们不一定有中央协调者，而是通过共享状态感知彼此发现。

这种模式适合协作研究和知识积累。风险是反应式循环：智能体不断响应彼此输出，消耗 token 但不产生收敛结果。

## 选型思路

可以用四个问题选择模式：

1. Worker 是否需要长期保持上下文？
   - 不需要：Orchestrator-Subagent。
   - 需要：Agent Teams。

2. 工作流是否可预测？
   - 可预测：Orchestrator 或固定 pipeline。
   - 不可预测：Message Bus。

3. 智能体是否需要彼此发现和协作？
   - 不需要：按任务分区。
   - 需要：Shared State。

4. 质量评估是否比生成更关键？
   - 是：Generator-Verifier。
   - 否：单智能体或简单委派即可。

默认建议从 Orchestrator-Subagent 开始，因为它协调成本最低、覆盖范围最广。只有当瓶颈真实出现时，再升级到团队、消息总线或共享状态。

## 与 Harness 的关系

多智能体协调是 Harness 设计的一部分。Harness 决定模型如何调用工具、管理上下文和接受反馈；多智能体模式决定多个模型如何分工和同步。

常见组合包括：

- Planner + Generator + Evaluator：适合从模糊需求到可运行产品。
- Orchestrator + 多个 Explorer：适合代码库调研和并行信息收集。
- Generator + Verifier + Browser：适合前端页面和视觉质量检查。
- Shared Knowledge Base + 多个任务 Agent：适合长期知识工程。

## 关键风险

多智能体系统最容易失败在三个地方：

1. **通信成本超过收益**：任务拆得太碎，综合信息比直接做更难。
2. **责任边界不清**：多个智能体改同一块内容，产生冲突。
3. **评价标准缺失**：看似有 verifier，实际只是复述生成者观点。

因此每次引入新智能体，都要能回答：它负责什么、不能碰什么、输出交给谁、怎样判断完成。

## 我的理解

多智能体协调的价值，不在“人多力量大”，而在让不同上下文各自保持清晰。复杂任务的难点常常不是推理能力不足，而是信息边界混乱。把任务拆给不同智能体，本质上是在拆注意力、拆状态、拆责任。

在个人项目里，我更倾向先用简单模式：主智能体负责总体推进，必要时并行派出调研、验证或文档整理角色。等项目复杂到需要持续 worker 或共享知识状态，再引入更重的团队模式。这样系统能保持轻巧，也能逐步演进。
