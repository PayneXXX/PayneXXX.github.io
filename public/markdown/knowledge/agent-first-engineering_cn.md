# 智能体优先工程

智能体优先工程不是“让 AI 帮我写几段代码”，而是一种软件工程重心的迁移：人类从亲手实现每一个细节，转向设计环境、约束、知识入口与反馈回路，让智能体能够在一个可观察、可验证、可持续维护的系统里完成工作。

它的核心判断是：当模型能力足够强，瓶颈就不再只是模型本身，而是模型所处的工作环境。一个通用聊天模型很难稳定完成复杂工程；但当它拥有文件系统、命令行、测试、文档地图、任务计划、回滚策略和评估标准，它就开始接近“工程参与者”。

## 核心转变

传统工程更像是人类直接操作代码库：理解需求、拆解任务、写代码、跑测试、修 bug。智能体优先工程则把这些动作拆成两层：

- 人类负责目标、边界、品味、验收标准和风险判断。
- 智能体负责搜索、实现、整理、迁移、验证和生成中间产物。

所以人类的工作不是消失，而是上移。一个好的工程师越来越像系统设计者、编辑、审稿人和环境构建者：他要知道如何让智能体不迷路，如何让它发现错误，如何让它在多小时任务里保持方向感。

## 四条实践路径

第一条路径是把代码仓库当成唯一记录系统。所有重要知识都应该落到仓库里，而不是散落在聊天记录、脑内记忆或临时文档中。架构规则、风格偏好、目录地图、执行计划和常见失败模式，都应该以 Markdown、schema、测试或 linter 的形式沉淀下来。智能体能读到什么，才真正拥有什么。

第二条路径是把“品味”转成可执行约束。比如什么样的 UI 不应该出现，什么文件不能被改，什么边界不能跨，什么验证必须运行。主观偏好无法直接被模型稳定继承，但可以被写成检查清单、示例、评分维度和自动化规则。

第三条路径是用反馈回路取代一次性提示。复杂任务不能指望一次 prompt 完成，而应该进入“计划、实现、检查、修正、复盘”的循环。测试、预览、截图、trace、日志和人工 review，都是让智能体从近似输出走向可用输出的反馈面。

第四条路径是持续维护智能体的工作环境。随着项目增长，文档会腐烂、规则会堆积、上下文会膨胀、工具会互相干扰。智能体优先工程需要定期清理：删掉过时规则，压缩入口文档，把常用流程沉淀成 Skills，把项目经验回填到知识库。

## 三层知识编码

面向智能体的知识不应该写成一篇巨大的说明书，而应该分层：

| 层级 | 作用 | 典型内容 |
| --- | --- | --- |
| 导航层 | 快速建立方向感 | ARCHITECTURE.md、AGENTS.md、目录地图 |
| 执行层 | 指导具体任务 | PLANS.md、验收标准、命令清单、约束 |
| 能力层 | 可复用专业知识 | Skills、模板、示例、领域知识卡 |

导航层回答“我在哪里”；执行层回答“这次要做什么”；能力层回答“遇到某类问题时应该如何做”。这三层合起来，才使智能体能从一次性助手变成可持续协作者。

## 与个人知识库的关系

智能体优先工程和个人知识库不是两件事。个人知识库负责把阅读、实践和复盘变成可复用知识；智能体优先工程负责把这些知识变成可调用的工作能力。

如果知识只供人类阅读，它最多提升人的理解速度；如果知识同时面向智能体组织，它就能进入任务执行过程：被检索、被加载、被引用、被更新，并在下一次任务里继续发挥作用。

这也是 Payne 当前 AI Building 工作流的关键：从信息检索出发，整理成 Wiki、知识图谱和信息地图，再进一步沉淀成全局 Skills 与项目专属 Skills，最终通过 MCP、CLI、工具链和执行计划进入真实项目。

## 判断一个系统是否 Agent-First

可以用几个问题判断：

1. 新智能体进入项目后，是否能在几分钟内知道架构边界？
2. 项目的关键规则是否写在仓库里，而不是只存在于人的脑子里？
3. 复杂任务是否有可恢复的执行计划，而不是只依赖长聊天记录？
4. 失败是否会产生可沉淀的经验，而不是下一次重新踩坑？
5. 输出质量是否有测试、预览、评分或人工 review 形成闭环？

如果这些问题的答案逐渐变成“是”，这个项目就开始从普通代码库转向智能体可协作的工程环境。

## 我的理解

智能体优先工程的本质，不是迷信智能体会自动完成一切，而是承认模型能力正在变成一种新的工程材料。材料本身有能力，也有边界；真正的价值来自人类如何设计它所处的结构。

未来的工程能力可能越来越体现为：能否把混乱需求整理成明确任务，能否把隐性经验写进系统，能否设计反馈回路，能否让人类知识和机器执行互相增强。这正是我想把个人网站中的 Knowledge 模块做成“知识文章”而不是简单卡片的原因：它应该展示一套正在形成的工程世界观。
